## **Projeto de Banco de Dados – Sistema de Vacinação**

### **1. Requisitos Funcionais**

Para desenvolver ou selecionar um sistema de banco de dados (SGBD) eficiente, é fundamental separar o que o sistema deve fazer (funcional) de como ele deve se comportar (não funcional).
Aqui está um resumo estruturado dos principais requisitos:

## 1. Requisitos Funcionais (O que o sistema faz)
Estão ligados às operações diretas de manipulação de dados e funcionalidades do software.

* Operações CRUD: O sistema deve permitir Criar, Ler, Atualizar e Deletar registros.
* Gestão de Transações (ACID): Garantir que as transações sejam Atômicas, Consistentes, Isoladas e Duráveis.
* Linguagem de Consulta: Suporte a uma linguagem padrão (como SQL para relacionais ou APIs específicas para NoSQL).
* Controle de Acesso: Criação de usuários, grupos e atribuição de permissões (quem pode ver ou editar o quê).
* Integridade de Dados: Implementação de restrições (constraints) como chaves primárias, chaves estrangeiras e campos obrigatórios.
* Backup e Recuperação: Ferramentas nativas para realizar cópias de segurança e restaurar dados em caso de falha.
* Processamento de Triggers e Procedures: Capacidade de executar scripts automáticos baseados em eventos. 
* 

------------------------------
## 2. Requisitos Não Funcionais (Como o sistema opera)
Referem-se à qualidade, desempenho e restrições técnicas.

* Desempenho (Performance): Tempo de resposta das consultas (ex: retornar buscas simples em menos de 100ms).
* Escalabilidade: Capacidade de lidar com o aumento de dados (Escalabilidade Vertical/Hardware ou Horizontal/Clusters).
* Disponibilidade: O tempo que o banco fica "online" (ex: disponibilidade de 99,9%).
* Segurança: Criptografia de dados em repouso e em trânsito, além de proteção contra injeção de SQL.
* Interoperabilidade: Capacidade de se conectar com diferentes linguagens de programação (Java, Python, PHP) e sistemas operacionais.
* Concorrência: Capacidade de suportar múltiplos usuários acessando e modificando os mesmos dados simultaneamente sem corrupção.
* Confiabilidade: Tolerância a falhas e consistência dos dados após quedas de energia ou erros de hardware.

# Sistema de vacina (Exemplo prático)

1. **Cadastrar paciente**
    - Permitir cadastro de paciente com nome, CPF, data de nascimento, telefone, status (ativo/inativo) e observações.
2. **Cadastrar vacina**
    - Cadastrar tipos de vacina (nome, descrição, fabricante, duração da imunidade em meses) em um catálogo.
3. **Registrar aplicação de vacina**
    - Registrar aplicação informando: paciente, vacina, clínica, data de aplicação, lote, dose (1ª, 2ª, 3ª, etc.) e responsável (profissional).
4. **Consultar histórico de vacinação do paciente**
    - Exibir lista de vacinas aplicadas a um paciente, com data, dose, clínica e responsável.
5. **Gerenciar clínicas**
    - Cadastrar, editar e listar clínicas (nome, endereço, responsável) utilizadas no sistema.
6. **Emitir relatórios básicos**
    - Gerar relatórios de:
        - vacinas aplicadas por período;
        - total de doses por tipo de vacina;
        - pacientes por clínica.

***

### **2. Requisitos Não Funcionais**

1. **Disponibilidade**
    - O sistema deve estar disponível em horário de atendimento de clínicas (pelo menos 8h por dia, 5 dias por semana).
2. **Segurança de dados**
    - CPF e dados de pacientes devem ser armazenados com controle de acesso.
    - Telas e relatórios não devem expor dados sensíveis sem autenticação.
3. **Performance**
    - Consulta de histórico de vacinação de um paciente deve retornar resultados em até 2 segundos, mesmo com até 100.000 registros.
4. **Integridade dos dados**
    - Não é permitido:
        - cadastrar dois pacientes com o mesmo CPF;
        - registrar aplicação sem paciente, vacina ou clínica válidos;
        - apagar dados de paciente com histórico sem auditoria.
5. **Auditoria e rastreabilidade**
    - O sistema deve registrar quem realizou cadastro/alteração de dados críticos (se possível, com módulo mínimo de log de auditoria).
6. **Compatibilidade**
    - Banco de dados em **MySQL 5.7 ou superior**, com suporte a servidor Linux/Windows.
7. **Usabilidade**
    - Principais funcionalidades (paciente, aplicação e histórico) devem ser acessíveis em até 3 telas.

### **3. Dicionário de Dados – Tabelas**


Um Dicionário de Dados (ou Metadados) é um repositório central que contém informações detalhadas sobre a estrutura e o conteúdo de um banco de dados. Ele não armazena os dados em si (como nomes ou valores), mas sim as regras e definições que os organizam.
Pense nele como o "manual de instruções" ou a "planta baixa" do banco de dados.

[DataSUS](https://datasus.saude.gov.br/wp-content/uploads/2023/04/Dicionario-de-dados_API-E-SUS-Notifica_13_12_22-1.pdf)
------------------------------

## O que ele contém?
Um bom dicionário de dados documenta os seguintes itens:

* Tabelas: Nome da tabela e sua finalidade.
* Colunas (Campos): Nome, tipo de dado (inteiro, texto, data) e tamanho.
* Restrições (Constraints): Se o campo pode ser nulo, se é único, etc.
* Chaves: Identificação de Chaves Primárias (PK) e Chaves Estrangeiras (FK).
* Relacionamentos: Como as tabelas se conectam entre si.
* Índices: Quais campos são usados para acelerar as buscas.

------------------------------
## Por que ele é importante?

   1. Padronização: Garante que todos os desenvolvedores usem os mesmos nomes e tipos para os mesmos dados.
   2. Facilidade de Manutenção: Ajuda novos membros da equipe a entenderem o sistema rapidamente.
   3. Integridade: Define as regras que evitam a entrada de dados incorretos.
   4. Segurança: Pode listar quem tem permissão para visualizar ou alterar cada dado.

------------------------------
## Tipos de Dicionário

* Ativo: É gerenciado automaticamente pelo Sistema Gerenciador de Banco de Dados (SGBD). Se você muda uma tabela, o dicionário atualiza sozinho (ex: as tabelas information_schema no MySQL ou PostgreSQL).
* Passivo: É um documento separado (como uma planilha ou PDF) mantido manualmente para fins de documentação.

## Documentação

Para um Sistema de Vacinação, os requisitos devem garantir tanto a eficiência do atendimento quanto a segurança rigorosa dos dados de saúde (dados sensíveis).

Abaixo, apresento um exemplo estruturado para o seu projeto:

## 1. Requisitos Funcionais (RF)
O que o banco de dados e o sistema devem permitir que o usuário faça:

* RF01 - Cadastro de Pacientes: O sistema deve registrar dados pessoais como Nome, CPF (ou Cartão SUS), data de nascimento e endereço.
* RF02 - Registro de Aplicação: O sistema deve permitir registrar cada dose aplicada, incluindo o lote da vacina, fabricante, data da aplicação e o profissional responsável.
* RF03 - Controle de Estoque: O banco de dados deve atualizar automaticamente a quantidade de doses disponíveis após cada registro de vacinação.
* RF04 - Agendamento de Doses: O sistema deve permitir o agendamento de doses futuras com base no intervalo mínimo recomendado pelo calendário de vacinação.
* RF05 - Emissão de Certificado: O sistema deve gerar um certificado de vacinação (frequentemente em PDF com QR Code) após a conclusão do esquema vacinal.
* RF06 - Notificação de Eventos Adversos: Deve haver um módulo para registrar e notificar possíveis reações adversas pós-vacinação.
* RF07 - Relatórios Epidemiológicos: Capacidade de gerar relatórios de cobertura vacinal por região, faixa etária ou tipo de imunizante.

------------------------------
## 2. Requisitos Não Funcionais (RNF)
Como o sistema deve se comportar em termos de qualidade e segurança:

* RNF01 - Segurança e Privacidade (LGPD): Os dados devem ser tratados como dados sensíveis, com criptografia em repouso e acesso restrito apenas a pessoal autorizado.
* RNF02 - Disponibilidade: O sistema deve estar disponível 99,9% do tempo, garantindo que as salas de vacina não fiquem paradas por falta de acesso aos dados.
* RNF03 - Interoperabilidade: O banco de dados deve ser capaz de trocar informações com a Rede Nacional de Dados em Saúde (RNDS) usando padrões como o HL7 FHIR.
* RNF04 - Desempenho: O registro de uma dose não deve levar mais do que 2 segundos para ser processado no banco de dados.
* RNF05 - Integridade: Uso de transações ACID para garantir que, se houver uma queda de energia no momento do registro, os dados não fiquem corrompidos.
* RNF06 - Auditoria (Logs): Toda alteração ou consulta a dados sensíveis de pacientes deve ser registrada em um log de auditoria inalterável.
