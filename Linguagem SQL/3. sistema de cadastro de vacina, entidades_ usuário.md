# Atividade Prática Sistema de cadastro de vacina, entidades: usuário, paciente, médico, enfermagem, técnico de enfermagem, aplicador de vacina.

Segue um **exemplo de modelagem de dados para cadastro de vacinação**, usando as entidades que você pediu e organizando-as de forma relacional. Em registros vacinais, é essencial identificar o cidadão atendido, o profissional que realizou a aplicação e os dados da dose aplicada, como vacina, lote e data.

## Ideia da modelagem.

Para evitar redundância, o melhor é tratar **médico, enfermagem, técnico de enfermagem e aplicador de vacina** como especializações de um cadastro base de profissional/usuário, já que todos são pessoas do sistema com login e perfil de atuação. O Ministério da Saúde destaca que o registro da vacinação deve identificar o profissional responsável pela aplicação e o cidadão atendido.

## Entidades propostas.

Você pode estruturar assim:

- **Usuário**: representa quem acessa o sistema, com login e senha.
- **Paciente**: representa quem recebe a vacina.
- **Profissional**: dados gerais do trabalhador de saúde.
- **Cargo_Profissional**: define se é médico, enfermagem, técnico de enfermagem ou aplicador de vacina.
- **Vacina**: cadastro da vacina.
- **Lote_Vacina**: controle de lote e validade.
- **Aplicacao_Vacina**: registro da dose aplicada no paciente.

Essa separação ajuda porque um mesmo profissional pode ter dados pessoais únicos e, ao mesmo tempo, estar vinculado a um cargo específico no sistema.

## Cardinalidades.

As cardinalidades ficam assim:

- **Usuário 1:1 Profissional** — um usuário pode estar associado a um profissional do sistema.
- **Cargo_Profissional 1:N Profissional** — um cargo pode estar ligado a vários profissionais.
- **Paciente 1:N Aplicacao_Vacina** — um paciente pode receber várias doses ao longo do tempo.
- **Profissional 1:N Aplicacao_Vacina** — um profissional pode aplicar várias vacinas.
- **Vacina 1:N Lote_Vacina** — uma vacina pode ter vários lotes.
- **Lote_Vacina 1:N Aplicacao_Vacina** — um lote pode aparecer em várias aplicações.

O registro de vacinação também costuma exigir informações como data da aplicação, profissional responsável, vacina administrada e identificação do usuário atendido.

## SQL para criar o banco.

Aqui está um exemplo em MySQL para criar o database do zero:

```sql
-- Criação de DATABASE do zero
CREATE DATABASE vacinacao_db;
USE vacinacao_db;

-- Criação de tabelas no DATABASE vacinacao_db
CREATE TABLE Usuario (
    id_usuario INT PRIMARY KEY AUTO_INCREMENT,
    nome_usuario VARCHAR(100) NOT NULL,
    login VARCHAR(50) NOT NULL UNIQUE,
    senha VARCHAR(255) NOT NULL,
    perfil VARCHAR(50) NOT NULL,
    status VARCHAR(20) DEFAULT 'ativo'
);

CREATE TABLE Paciente (
    id_paciente INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(120) NOT NULL,
    cpf VARCHAR(14) UNIQUE,
    cns VARCHAR(20) UNIQUE,
    data_nascimento DATE NOT NULL,
    sexo CHAR(1),
    telefone VARCHAR(20),
    endereco VARCHAR(150)
);

CREATE TABLE Cargo_Profissional (
    id_cargo INT PRIMARY KEY AUTO_INCREMENT,
    nome_cargo VARCHAR(60) NOT NULL UNIQUE
);

CREATE TABLE Profissional (
    id_profissional INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(120) NOT NULL,
    cpf VARCHAR(14) NOT NULL UNIQUE,
    cns VARCHAR(20),
    registro_conselho VARCHAR(30),
    id_cargo INT NOT NULL,
    id_usuario INT UNIQUE,
    FOREIGN KEY (id_cargo) REFERENCES Cargo_Profissional(id_cargo),
    FOREIGN KEY (id_usuario) REFERENCES Usuario(id_usuario)
);

CREATE TABLE Vacina (
    id_vacina INT PRIMARY KEY AUTO_INCREMENT,
    nome_vacina VARCHAR(100) NOT NULL,
    fabricante VARCHAR(100),
    descricao VARCHAR(200)
);

CREATE TABLE Lote_Vacina (
    id_lote INT PRIMARY KEY AUTO_INCREMENT,
    numero_lote VARCHAR(50) NOT NULL UNIQUE,
    data_validade DATE NOT NULL,
    quantidade_disponivel INT NOT NULL,
    id_vacina INT NOT NULL,
    FOREIGN KEY (id_vacina) REFERENCES Vacina(id_vacina)
);

CREATE TABLE Aplicacao_Vacina (
    id_aplicacao INT PRIMARY KEY AUTO_INCREMENT,
    data_aplicacao DATE NOT NULL,
    dose VARCHAR(30) NOT NULL,
    via_aplicacao VARCHAR(50),
    local_aplicacao VARCHAR(50),
    observacao VARCHAR(255),
    id_paciente INT NOT NULL,
    id_profissional INT NOT NULL,
    id_lote INT NOT NULL,
    FOREIGN KEY (id_paciente) REFERENCES Paciente(id_paciente),
    FOREIGN KEY (id_profissional) REFERENCES Profissional(id_profissional),
    FOREIGN KEY (id_lote) REFERENCES Lote_Vacina(id_lote)
);
```

## Consulta útil

Para mostrar o histórico de vacinação do paciente:

```sql
-- Primeira consulta depois de inserir dados em tabelas no DATABASE vacinacao_db
SELECT 
    p.nome AS paciente,
    v.nome_vacina,
    a.dose,
    a.data_aplicacao,
    pr.nome AS profissional,
    c.nome_cargo,
    l.numero_lote
FROM Aplicacao_Vacina a
JOIN Paciente p ON a.id_paciente = p.id_paciente
JOIN Profissional pr ON a.id_profissional = pr.id_profissional
JOIN Cargo_Profissional c ON pr.id_cargo = c.id_cargo
JOIN Lote_Vacina l ON a.id_lote = l.id_lote
JOIN Vacina v ON l.id_vacina = v.id_vacina;
```
