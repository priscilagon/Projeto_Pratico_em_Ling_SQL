# Modelagem DER.

O DER (Diagrama de Entidade e Relacionamento) é a ferramenta visual utilizada para a modelagem conceitual de bancos de dados. Ele descreve a estrutura lógica dos dados antes de qualquer implementação técnica. 

Aqui está um resumo prático para você montar ou revisar sua modelagem:

1. Componentes Principais
Entidade (Retângulo): Representa um objeto do mundo real (ex: CLIENTE, PRODUTO, PEDIDO).
Atributo (Elipse/Círculo): São as características da entidade (ex: Nome, CPF, Preço).
Relacionamento (Losango): Conecta as entidades e descreve como elas interagem (ex: Cliente compra Produto). 
2. Cardinalidade (O "Pulo do Gato")
Define quantos itens de uma entidade podem se associar a itens de outra: 
1:1 (Um para Um): Um usuário possui apenas um currículo.
1:N (Um para Muitos): Um cliente pode fazer vários pedidos, mas cada pedido pertence a apenas um cliente.
N:N (Muitos para Muitos): Vários alunos podem se matricular em várias disciplinas. 
3. Passo a Passo Simples
Identifique as Entidades: Quais são os "substantivos" centrais do sistema?
Defina os Relacionamentos: Como esses substantivos se relacionam entre si ("verbos")?
Adicione os Atributos: O que você precisa saber sobre cada entidade?
Marque a Chave Primária (PK): O atributo único que identifica cada registro (ex: ID_Cliente). 

# Regras de negócio principais do (Sistema de vacina)

1. Regras de Paciente e Elegibilidade
Identificação Única: Cada paciente deve ser identificado por um número único (CPF ou Cartão SUS).
Faixa Etária: Algumas vacinas são exclusivas para idades específicas (ex: BCG para recém-nascidos, Gripe para idosos).
Histórico de Reações: O sistema deve registrar se o paciente teve efeitos colaterais em doses anteriores de um fabricante específico.
2. Regras de Aplicação e Protocolo
Esquema Vacinal (Doses): Uma vacina pode exigir uma dose única ou múltiplas doses (D1, D2, Reforço).
Intervalo Mínimo: O sistema não deve permitir a aplicação da D2 se o intervalo de tempo mínimo (ex: 21 dias) em relação à D1 não tiver sido cumprido.
Intercambialidade: Definir se o paciente pode tomar doses de fabricantes diferentes (mix-and-match) ou se deve obrigatoriamente seguir a mesma marca. 
3. Regras de Inventário (Lote e Validade)
Rastreabilidade Total: Toda aplicação deve estar vinculada a um Número de Lote e data de validade.
Baixa Automática: Ao registrar uma vacinação, o sistema deve subtrair automaticamente 1 unidade do estoque da unidade de saúde.
Validade: O sistema deve bloquear a seleção de lotes que já expiraram. 
4. Regras de Profissionais e Local
Responsabilidade: Toda vacina deve registrar o profissional (Enfermeiro/Técnico) que realizou a aplicação (via registro profissional/coren).
Unidade de Saúde: O sistema deve saber em qual posto físico a dose foi aplicada para controle geográfico.

## Leitura do diagrama (Comentada).

Um **paciente** pode ter várias aplicações de vacina, por isso a relação entre **Paciente** e **Aplicacao_Vacina** é **1:N**. Um **profissional** também pode realizar várias aplicações, enquanto cada aplicação é registrada por um único profissional responsável. Cada **lote de vacina** pertence a uma única vacina, e uma vacina pode possuir vários lotes para controle de estoque e rastreabilidade.

## Regra das especializações.

Listou **médico, enfermagem, técnico de enfermagem e aplicador de vacina**, a melhor modelagem é usar uma entidade **Cargo_Profissional** ligada à entidade **Profissional**, evitando repetição de estrutura para cada tipo de trabalhador. Assim, o sistema fica mais simples de manter e mais escalável caso surjam novos perfis profissionais depois.
